# 技能开发尖锐批判（Claude Skills）

本文件针对“技能开发”的普遍误区进行批判与矫正，目标是把 Skills 从“项目维基/教程”强制拉回“速查手册”，提升检索命中率与可操作性，切实实现降本增效。

## 十宗罪（共性失败模式）

- 定位错位：将 Skill 写成教程或百科。症状：长背景、流程解说、教学口吻。
- 篇幅失控：超过 500 行或信息密度低，关键信息被淹没。
- Frontmatter 失职：缺少触发性 description 与关键词；未说明“何时使用”。
- 信息无层级：核心与细节混杂；未采用“渐进式披露”。
- 职责交叉：技能之间重复/重叠，边界模糊，导致检索冲突。
- 通用知识侵蚀：Pandas/Tailwind/REST 等基础教程混入，稀释项目特异性。
- 模板与技能混堆：长代码示例直接塞进 SKILL.md，导致肥大与维护困难。
- 未来性/假设性内容：与当前代码不一致或尚未落地，误导调度与实现。
- 元数据不一致：路线图、目录、技能名称不统一，造成定位错误。
- 触发词贫瘠：缺少高频联想词，检索模型难以命中正确 Skill。

## 重构原则（硬约束）

- Cheat Sheet First：只保留“立刻可用”的信息，拒绝教程化叙事。
- 500 行硬上限：能更短则更短；超出内容全部外置到 reference/。
- 最小完备 Frontmatter：包含 name、description（触发词+使用场景）、links。
- 渐进式披露：主文件给速查要点；背景、长示例与细则放到 reference/。
- 项目现实优先：仅反映当前代码与工作流，删去假设与过时内容。
- 职责分离：ETL vs 分析、规范 vs 模板、契约 vs 测试，边界清晰。

## 推荐 SKILL.md 结构（可复制执行）

1. Frontmatter：`name`，`description`（含触发词与“何时用”），`links`
2. 核心速查：关键指标/接口列表/方法签名/字段名（极简要点即可）
3. 使用要点：2–5 条“如何调用/如何避免坑”的准则
4. DOs / DON'Ts：条目化约束，强调项目特异性规则
5. Reference：链接到 reference/ 与相关文档，不在主文件塞长文

## 触发词与联想词库（示例）

- KPI、筛选字段、方法签名、ETL、数据清洗、错误码
- Pinia Store、ECharts、useChart、响应体、分页、排序、缓存
- 数据字典、校验规则、BEM、CSS 变量、主题色、部署脚本、日志查看

说明：将以上词汇注入 `description` 与核心段落中，显著提升检索命中率与调度准确性。

## 质量度量（衡量重构成效）

- 压缩率：重构前后行数变化（目标≥ 40%）。
- 命中率：面向 3 个常见任务的检索成功率（目标≥ 80%）。
- 冲突密度：技能间重复条目占比（目标→ 低）。

## 审查清单（发布前必勾选）

- 是否是速查而非教程？
- 是否 ≤ 500 行？
- `description` 是否包含触发词与使用场景？
- 长代码/示例是否外置到 reference/？
- 是否与代码库现状一致？
- 职责边界是否清晰、无重复？

## 迁移策略（从臃肿到可用）

- 分拆：把混杂内容拆成子技能或 reference 文件。
- 链接化：移除长段落，保留必要链接与锚点。
- 清理：删除未来性、过时或与项目无关的内容。

## 维护工作流（保持长期健康）

- 版本约定：Skill 使用 `vX.Y`，变更日志简短记录要点与影响面。
- 变更触发：当代码路径/API/指标口径变化时同步更新 Skill。
- 审批机制：PR 模板包含本审查清单，拒绝未经审查的教程化补丁。

## 结论

Skills 是“助攻工具”，不是“百科全书”。以“速查手册”的心态编写与维护，才能真正提升协作效率、降低对话成本，并确保每次调用都准确落在项目的现实与关键路径之上。
